# M7 — System page

**Status:** Not started  
**Validates:** Owner sees network/disk/uptime; remote SSH toggle defaults off; reboot requires confirmation.

## Goal

System visibility and guarded power/SSH controls from dashboard.

## Prerequisites

- [M5](M5.md) dashboard with routing
- [M1](M1.md) SSH and systemd baseline

## Implementation steps

### 1. API (`frogswork_api/system/`)

| Endpoint | Purpose |
|----------|---------|
| `GET /api/system/info` | Hostname, IP, uptime, disk free/total, API version |
| `GET /api/system/ssh` | `{ remote_enabled: bool }` |
| `POST /api/system/ssh` | Enable/disable remote SSH (admin only) |
| `POST /api/system/reboot` | Schedule reboot (confirm body) |
| `POST /api/system/shutdown` | Schedule shutdown (confirm body) |

### 2. SSH toggle implementation

- **Default off** on fresh install
- Enable: adjust `sshd_config` or drop-in in `/etc/ssh/sshd_config.d/frogswork.conf`
- `systemctl reload ssh` after change
- Clearly label in UI: "Allow remote support SSH"
- No baked-in KorraOne keys

### 3. Reboot/shutdown

- Use `systemd-run` or `shutdown` with delay
- Require `{ "confirm": true }` in POST body
- Return plain-language warning in response

### 4. Dashboard — **System** page

- Network info (IP, hostname `frogswork.local`)
- Disk usage bar
- Uptime
- SSH toggle switch
- Reboot / shutdown buttons with modal confirm
- Link to download helper app (prepare for M8)

### 5. Security

- All routes require dashboard admin auth
- Audit log optional (defer if scope creep)

## Verification

1. System page shows correct disk and network data
2. SSH toggle off by default; enable → SSH from laptop works; disable → blocked
3. Reboot button reboots NUC; services come back automatically (M1 enable)
