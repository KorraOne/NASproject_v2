# M5 — Dashboard UI

**Status:** Not started  
**Validates:** Owner completes full journey in browser — setup, users, folders, storage — without curl.

## Goal

React pages that consume the existing API. No duplicate business logic in the frontend.

## Prerequisites

- [M2](M2.md)–[M4](M4.md) complete — all owner APIs available
- Dashboard builds and deploys via `02-deploy-app.sh`

## Implementation steps

### 1. API client (`dashboard/src/api/`)

- Typed fetch wrapper with auth cookie/token handling
- Modules: `setup.ts`, `auth.ts`, `users.ts`, `folders.ts`, `storage.ts`

### 2. Routing

- React Router (add dependency)
- Redirect to `/setup` if `setup_complete` is false
- Protected routes require admin login

### 3. Pages (`dashboard/src/pages/`)

| Page | Features |
|------|----------|
| **Setup** | Admin password, timezone, submit wizard |
| **Login** | Dashboard admin sign-in |
| **Users** | List/create/edit/delete file users; quota; super-user toggle; inline folder permissions |
| **Folders** | List/create/edit shared folders; bulk permission matrix |
| **Storage** | Disk usage overview (API from system or dedicated endpoint) |

Defer Snapshots and System pages to M6/M7 (show nav disabled or omit until ready).

### 4. UX principles

- Plain language — "Snapshot" not "btrfs subvolume"
- FrogsWork branding — title, simple layout, accessible forms
- Error messages from API `detail` field

### 5. Dev workflow

```bash
cd dashboard
npm run dev   # proxies /api to frogswork.local per vite.config.ts
```

### 6. Deploy

- `npm run build` → `dist/` copied on NUC by install script
- Verify `http://frogswork.local` serves SPA and API routes work

## Verification

1. Fresh NUC (or reset DB): open dashboard → complete setup wizard
2. Create two file users, assign folder permissions
3. Storage page shows disk usage
4. Logout/login works
